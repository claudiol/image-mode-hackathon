# GitLab Image Mode Deployment on AWS EC2

This Ansible playbook automates the deployment of a self-managed GitLab instance on an AWS EC2 host for the Image Mode lab. It prepares the RHEL system, installs GitLab, configures HTTPS and the integrated GitLab Container Registry, enables Red Hat IdM LDAP authentication, installs the IdM CA into GitLab's embedded trust store, initializes IdM administrators as GitLab administrators before first login, creates the default `redhat-consulting` and `image-mode` groups, and validates the resulting GitLab services and LDAP configuration.

---

## 1. Architectural Highlights & Automated Operations

The playbook performs a complete single-node GitLab deployment workflow for an AWS-hosted RHEL system.

* **Terraform-Generated Infrastructure Integration:** Consumes AWS, DNS, TLS, Secrets Manager, and host metadata generated by Terraform.
* **GitLab Omnibus Installation:** Installs and configures a self-managed GitLab instance on the target EC2 host.
* **Persistent Hostname Configuration:** Configures the GitLab FQDN and preserves it across reboots.
* **HTTPS Configuration:** Uses a pre-staged certificate and private key for the GitLab web interface.
* **Integrated Container Registry:** Enables the GitLab Container Registry on a dedicated HTTPS endpoint and configurable registry port.
* **Red Hat IdM LDAP Authentication:** Configures GitLab to authenticate users against Red Hat IdM over LDAPS.
* **Embedded GitLab CA Trust:** Copies the IdM CA into `/etc/gitlab/trusted-certs` and runs `gitlab-ctl reconfigure` so GitLab's embedded OpenSSL stack trusts IdM.
* **LDAP Just-in-Time Access:** Allows regular IdM users matching the configured LDAP filter to sign in and receive GitLab accounts automatically.
* **Administrator Bootstrap Without First Login:** Queries the IdM administrator group and initializes each current IdM administrator as a linked GitLab LDAP user before any interactive login occurs.
* **Default Group Creation:** Creates the `redhat-consulting` and `image-mode` GitLab groups.
* **Administrator Group Ownership:** Assigns bootstrapped IdM administrators as Owners of both default groups.
* **Project Creation Enablement:** Configures both default groups so Developer-level members can create projects in those namespaces.
* **Secrets Manager Integration:** Retrieves the GitLab root password, LDAP bind password, and related secrets from AWS Secrets Manager.
* **Post-Install Validation:** Runs GitLab service, application, LDAP, TLS, configuration, permission, and administrator bootstrap checks.

## 2. Prerequisites & Sizing Requirements

### Target Instance Baseline

* **Operating System:** Red Hat Enterprise Linux 9 or later
* **Instance Type:** `m6i.large`, `m6a.large`, or larger recommended for lab use
* **CPU:** 2 vCPUs minimum
* **Memory:** 8 GiB RAM minimum
* **Storage:** 80+ GiB root volume recommended
* **User:** `ec2-user` must exist and have sudo privileges
* **Network:** The instance must reach:
  * Red Hat package repositories
  * GitLab package repositories
  * AWS Secrets Manager
  * Red Hat IdM over LDAPS
  * Route 53 or IdM-backed DNS
* **DNS:** The GitLab FQDN and registry FQDN must resolve to the GitLab instance
* **TLS:** The GitLab server certificate and private key must be staged before GitLab reconfiguration
* **IdM Enrollment:** The GitLab host must already be enrolled in IdM so `/etc/ipa/ca.crt` exists

### AWS Security Group Requirements

Allow inbound access to the required GitLab service ports from trusted networks:

* **TCP 22:** SSH administration
* **TCP 80:** HTTP redirect to HTTPS
* **TCP 443:** GitLab HTTPS
* **TCP 5050:** GitLab Container Registry, or the configured registry port

For internal-only services, restrict access to the VPC CIDR or other trusted sources.

### IAM Instance Profile Requirements

The GitLab EC2 instance profile should allow only the actions required by the deployment.

Typical permissions include:

* `secretsmanager:GetSecretValue`
* `secretsmanager:DescribeSecret`
* `acm:DescribeCertificate`
* `acm:ExportCertificate`, when the certificate is exported on the host
* `s3:GetObject`, when installation assets are staged from S3

Where possible, scope access to the environment-specific secret prefix and certificate tags.

## 7. GitLab LDAP Configuration

The playbook renders `/etc/gitlab/gitlab.rb` with IdM LDAP enabled.

Representative configuration:

```ruby
gitlab_rails['ldap_enabled'] = true
gitlab_rails['prevent_ldap_sign_in'] = false

gitlab_rails['ldap_servers'] = {
  'main' => {
    'label' => 'Red Hat IdM',
    'host' => 'idm-1.lab.example.com',
    'port' => 636,
    'uid' => 'uid',

    'bind_dn' => 'uid=gitlab-bind,cn=users,cn=accounts,dc=lab,dc=example,dc=com',
    'password' => 'REDACTED',

    'encryption' => 'simple_tls',
    'verify_certificates' => true,
    'timeout' => 10,

    'tls_options' => {
      'ca_file' => '/etc/gitlab/trusted-certs/red-hat-idm-ca.crt'
    },

    'active_directory' => false,
    'allow_username_or_email_login' => true,
    'lowercase_usernames' => true,
    'block_auto_created_users' => false,

    'base' => 'cn=users,cn=accounts,dc=lab,dc=example,dc=com',
    'user_filter' => '(objectClass=person)',

    'attributes' => {
      'username' => ['uid'],
      'email' => ['mail'],
      'name' => 'cn',
      'first_name' => 'givenName',
      'last_name' => 'sn'
    }
  }
}

gitlab_rails['omniauth_allow_single_sign_on'] = ['ldapmain']
gitlab_rails['omniauth_block_auto_created_users'] = false
```

The LDAP provider identifier is:

```text
ldapmain
```

This provider value must also be used when pre-creating LDAP identities during administrator bootstrap.

## 8. IdM CA Trust Model

GitLab Omnibus uses its own embedded OpenSSL trust store.

Adding the IdM CA only to the RHEL system trust store is not sufficient.

The playbook must:

1. Verify `/etc/ipa/ca.crt`.
2. Create:

```text
/etc/gitlab/trusted-certs
```

3. Copy the IdM CA to:

```text
/etc/gitlab/trusted-certs/red-hat-idm-ca.crt
```

4. Run:

```bash
gitlab-ctl reconfigure
```

5. Verify that GitLab created the corresponding hashed link under:

```text
/opt/gitlab/embedded/ssl/certs
```

6. Verify LDAPS with:

```bash
/opt/gitlab/embedded/bin/openssl s_client \
  -connect idm-1.lab.example.com:636 \
  -servername idm-1.lab.example.com \
  -CAfile /etc/gitlab/trusted-certs/red-hat-idm-ca.crt \
  -verify_hostname idm-1.lab.example.com \
  -verify_return_error
```

Expected result:

```text
Verify return code: 0
```

## 9. Administrator Bootstrap Without First Login

The playbook initializes the current IdM administrators before anyone signs in interactively.

The bootstrap process:

1. Queries the configured IdM administrator group.
2. Extracts direct user member DNs.
3. Queries each user for:
   * `uid`
   * `cn`
   * `mail`
   * full LDAP DN
4. Creates or locates the corresponding GitLab account.
5. Creates or repairs the `ldapmain` identity.
6. Sets:

```text
admin = true
```

7. Creates the default groups:
   * `redhat-consulting`
   * `image-mode`
8. Adds every bootstrapped IdM administrator as Owner of both groups.
9. Verifies the user, LDAP identity, administrator flag, and group memberships.

Example IdM administrator definition constructed by the playbook:

```yaml
gitlab_idm_admin_users:
  - username: "phil"
    name: "Phil Duncan"
    email: "phil@example.com"
    dn: >-
      uid=phil,cn=users,cn=accounts,dc=lab,dc=example,dc=com
```

The full LDAP DN is stored as the GitLab identity `extern_uid`.

After bootstrap, the administrator can sign in with IdM credentials and GitLab resolves the existing LDAP-linked account rather than creating a new account.

No manual first-login bootstrap is required.

## 10. Default GitLab Groups

The deployment creates two top-level groups:

```text
redhat-consulting
image-mode
```

Recommended group settings:

* **Visibility:** Internal
* **Project creation level:** Developer
* **Bootstrapped IdM administrator access:** Owner

This allows users who are later granted Developer membership to create projects in those group namespaces.

The playbook intentionally does not grant all IdM users automatic membership unless a separate group-mapping or lifecycle process is added.

## 15. Important Design Notes

### GitLab Uses an Embedded Trust Store

GitLab Omnibus does not rely exclusively on the RHEL system trust store for LDAP TLS verification.

The IdM CA must exist under:

```text
/etc/gitlab/trusted-certs
```

and GitLab must be reconfigured afterward.

### Do Not Disable Certificate Verification

Avoid:

```ruby
'verify_certificates' => false
```

The correct solution is to install the IdM CA and leave verification enabled.

### LDAP Provider Must Match the Server Key

For this configuration:

```ruby
gitlab_rails['ldap_servers'] = {
  'main' => {
  }
}
```

the corresponding GitLab provider is:

```text
ldapmain
```

The administrator bootstrap must use the same provider.

### Pre-Created LDAP Accounts Do Not Need a Manual First Login

The playbook creates the GitLab account and LDAP identity before the user signs in.

The user's IdM password is not stored in GitLab and is not used during bootstrap.

A random local password is generated only to satisfy GitLab account creation requirements. Normal authentication continues through IdM LDAP.

### Current Administrator Membership Is Bootstrapped

The one-time deployment initializes the IdM administrators that exist when the playbook runs.

IdM administrators created later require a separate lifecycle process if they must be elevated automatically.

### Direct Group Membership

The current LDAP query extracts direct user members from the IdM administrator group.

If the IdM group contains only nested groups, the query or group design must be updated to expand nested membership.

### Email Attribute Is Required

Every bootstrapped IdM administrator must have a valid `mail` attribute.

GitLab requires a usable email address when creating the account.

## 16. Idempotency Behavior

The deployment is designed to be rerunnable.

The playbook:

* Reuses existing GitLab directories.
* Reuses correctly installed certificates.
* Re-renders `/etc/gitlab/gitlab.rb` only when content changes.
* Runs `gitlab-ctl reconfigure` through a handler.
* Reuses existing GitLab users.
* Reuses existing `ldapmain` identities.
* Promotes users only when needed.
* Reuses existing default groups.
* Updates group settings only when needed.
* Creates group memberships only when missing.
* Changes group access only when incorrect.
* Optionally demotes removed IdM administrators when removal reconciliation is enabled.

Recommended default:

```yaml
gitlab_idm_admin_reconcile_removals: false
```

This avoids unintentionally demoting a GitLab administrator whose access was granted independently.

## 17. Common Failure Points

### IdM CA Source Certificate Is Missing

Error:

```text
The Red Hat IdM CA certificate was not found at /etc/ipa/ca.crt
```

Confirm the GitLab host is enrolled in IdM:

```bash
sudo test -s /etc/ipa/ca.crt
sudo ipa-client-install --unattended
```

### GitLab Trusted Certificates Directory Does Not Exist

Confirm the playbook creates:

```text
/etc/gitlab/trusted-certs
```

with:

```yaml
state: directory
owner: root
group: root
mode: "0755"
```

Use single-line path variables or normalize them with `trim`.

### LDAP Certificate Verification Fails

Error:

```text
certificate verify failed
self-signed certificate in certificate chain
```

Verify:

```bash
sudo ls -l /etc/gitlab/trusted-certs
sudo gitlab-ctl reconfigure
sudo gitlab-rake gitlab:ldap:check
```

### LDAP Bind Password Retrieval Fails

Verify the GitLab EC2 role can read:

```text
<secret_prefix>/idm/admin_password
```

or the dedicated LDAP bind secret.

Check:

```bash
aws secretsmanager get-secret-value \
  --secret-id image-mode-lab/idm/admin_password \
  --region us-east-2 \
  --query SecretString \
  --output text
```

### No IdM Administrator Members Found

Error:

```text
IdM group exists, but no direct IdM user members were found
```

Check the group:

```bash
ipa group-show idm-admins --all
```

Ensure it contains direct user members or update the playbook to expand nested groups.

### IdM Administrator Has No Mail Attribute

Check:

```bash
ipa user-show <username> --all
```

Add an email address:

```bash
ipa user-mod <username> --email=<username>@example.com
```

### LDAP Identity Conflict

The playbook stops if the IdM username, email, or LDAP DN resolves to different GitLab users.

Inspect existing users and identities before forcing changes.

### Registry Is Unreachable

Verify:

```bash
sudo firewall-cmd --list-ports
sudo gitlab-ctl status registry
curl -k https://gitlab-1.lab.example.com:5050/v2/
```

### GitLab Configuration Permissions Are Incorrect

Required:

```text
/etc/gitlab/gitlab.rb              root:root 0600
/etc/gitlab/trusted-certs/*.crt    root:root 0644
/etc/gitlab/ssl/*.key              root:root 0600
```

## 19. Post-Install Access

GitLab web interface:

```text
https://gitlab-1.lab.example.com
```

GitLab Container Registry:

```text
https://gitlab-1.lab.example.com:5050
```

Authentication behavior:

* **IdM administrators existing during bootstrap**
  * Already initialized in GitLab
  * Already linked to `ldapmain`
  * GitLab instance administrators
  * Owners of `redhat-consulting`
  * Owners of `image-mode`
* **Other IdM users**
  * Created through LDAP just-in-time provisioning on first successful sign-in
  * Not automatically added to either default group unless another workflow grants membership

The local GitLab root account remains available for break-glass administration using the root password stored in AWS Secrets Manager.
